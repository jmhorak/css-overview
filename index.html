Browser Basics

How It Works

After any page is loaded, browsers immediately start parsing HTML elements.

  • Scripts encountered are executed immediately
  • Mark-up is parsed and organized into the DOM Tree
  • CSS is parsed into buckets based on the selector

DOM and Render Trees

The DOM tree is the result of parsing the HTML that comes back from the network layer.

The DOM Tree is combined with the CSS to form the render tree. This is ultimately what is painted on-screen.

There is a 1:1 relationship between HTML elements and DOM elements. There is not a 1:1 relationship between DOM elements and render elements.

DOM tree
  • html
    • head
      • title
      • style
    • body
      • h1
      • div
        • span
        • img
        • a
      • div
        • text block
CSS span { display: none; }
Render tree
  • viewport
    • scrolling area
      • h1 block
      • div block
        • img block
        • a text block
      • div block
        • text line
          • text block
        • text line
          • text block

Painting

For better user experience, the rendering engine will try to display contents on the screen as soon as possible. It will not wait until all HTML is parsed before starting to build and layout the render tree. Parts of the content will be parsed and displayed, while the process continues with the rest of the contents that keeps coming from the network.
Source: How Browsers Work

Painting

The rendering engine paints the render tree elements in order based on the stacking context.

The stacking context encapsulates z-index, element flow, and position within the mark-up.

The stacking order within a context for a block renderer is:

  • background color
  • background image
  • border
  • children
  • outline
Source: How Browsers Work, W3C - Stacking contexts

How Are Conflicting Styles Resolved?

The rendering engine may have multiple styles it needs to apply to a particular element.

div { background-color: #00FF00; }
div { background-color: #202020; }
Should the background be green or gray?

Resolution Characteristics

Resolving conflicts requires looking at several pieces of information

  • Rule origin - Where did the rule originate?
  • Specificity - How specific is the rule?
  • Importance - Was the rule declared as !important?
  • Order - What order were the rules loaded?
Further Reading: The Cascade

Resolution Steps

Resolving CSS properties is done in four steps

  1. For a given property, find all declarations that apply to a specific element.
  2. Sort the declarations according to their levels of importance, and origins.
  3. Sort declarations with the same level of importance and origin by selector specificity.
  4. Finally, if declarations have the same level of importance, origin, and specificity, sort them by the order in which theyâ€™re specified; the last declaration wins.
Source: The Cascade

Rule Origin

The origin of a CSS rule refers to the type of stylesheet where it appeared.

Rules are imported from three sources

User Agent The default stylesheet for the device rendering the output. This is usually a web browser but may be a phone, TV, or household appliance
User Stylesheet Most desktop browsers allow users to provide their own custom defaults in the user stylesheet.
Author Stylesheet Any rule imported via a link or style element or in an element's style attribute.

How Origin is Applied

CSS rule declarations are sorted by origin in the following order from lowest to highest

  1. User agent declarations
  2. Normal declarations in user style sheets
  3. Normal declarations in author style sheets
  4. Important declarations in author style sheets
  5. Important declarations in user style sheets
Further Reading: The Cascade

Specificity

More specific rules are applied over less specific ones. But how is specificity calculated?

Rules are split into 4 categories, a, b, c, d.

  • a = 1 if the declaration is inline rather than on a selector, else a = 0
  • b = the count of the number of ID attributes
  • c = the count of other attributes (like class names) and pseudo-classes
  • d = the count of element names and pseudo-elements

Concatenating the four counts (in a hex base) gives the specificity

Source: Specificity

Units of Measurement

Font-Relative Lengths

Aside from rem (which refers to the font-size of the root element), the font-relative lengths refer to the computed font metrics of the element on which they are used. The exception is when they occur in the value of the font-size property itself, in which case they refer to the font metrics of the parent element (or the font metrics corresponding to the initial values of the font property, if the element has no parent).
Source: Font-Relative Lengths

Font Relative Lengths

1.0em Equal to the computed value of the font-size
1.0ex Equal to the height of the lowercase "x"
1.0ch Equal to the height of 0
1.0rem Equal to the computed value of the font-size of the root element 
h1 { line-height: 1.2em; }

The line height of h1 elements will be 20% greater than the font size of the h1 element 

h1 { font-size: 1.2em; }

The font-size of h1 elements will be 20% greater than the inherited font size

Source: Font Relative Lengths

Absolute Lengths

Absolute length units are fixed in relation to each other and anchored to some physical measurement.

Some common units of measurement:

cm centimeters
in inches - defined as 2.54cm
px pixel - defined as 1/96th of an inch or the size of the device's reference pixel
pt points - defined as 1/72nd of an inch

In practice, the best unit of measurement depends on the media.

Pixels make sense on screen. Centimeters and inches make sense in printed media.

Source: Absolute Lengths

Percentages

Percentage values are always relative to some other value. Properties that allow percentages implicitly define the value to which the percentage refers.

Source: Percentages

Viewport-Relative Units

CSS3 introduces viewport-relative units. These are supported in Chrome and IE.

Percentages are relative to a value on an ancestor element. Viewport-relative units are always in relation to the viewport size.

vh 1% of the viewport height
vw 1% of the viewport width
vm The smaller of vh and vw
Further Reading: Viewport-Sized Typography

Block Attributes

Some interesting properties of block-level elements

  • Blocks start on a newline
  • Blocks expand to use all available horizontal space
  • Blocks expand vertically to fully contain in-flow child elements as part of their footprint.
  • Blocks create a new block-formatting context (more on this later)
  • The top and bottom margins between block elements collapse

The top and bottom margin between sibling block elements are combined into a single margin. The resulting margin is the larger of the two being combined.

Colors, Shadows, Gradients

Colors

Three primary ways to specify color in CSS:

  • Named Color
  • RGB
  • HSL

These are all representations of the same color 

h1 { color: lightslategray; }
h2 { color: #778899; }
h3 { color: #789; }
h4 { color: rgb(119, 136, 153); }
h5 { color: hsl(140, 34, 128); }
Further Reading: Colors, Mother-effing HSL

Transparency

RGB and HSL colors may include transparency through an alpha channel. This is specified using the rgba and hsla functions

Alpha values range from 0 to 1. The higher the value the more opaque.

The transparent named color is fully transparent.

background-color: rgba(0, 0, 255, 1); /* full opaque */
background-color: rgba(0, 0, 255, .5);/* 50% opaque */
background-color: rgba(0, 0, 255, .1); /* 10% opaque */
background-color: transparent; /* Fully transparent */

Linear Gradients

The CSS linear gradient function creates an image on the fly displaying the gradient between the supplied colors.

Gradients may have multiple color-stops. Color stops are points at which the gradient reaches a specific color. 

.myAwesomeGradient {
  linear-gradient(45deg, red, orange 50%, green, blue 80%, purple);
}
Further Reading: Linear Gradients

Radial Gradients

The radial gradient function creates an image on the fly displaying a gradient of colors radiating away from an origin.

Radial gradients may appear as circles or ellipses. 

.myOtherAwesomeGradient {
 radial-gradient(circle, red, orange 50%, green, blue 80%, purple);
}

Repeating Gradients

Linear and radial gradients can be set as repeating. The color stops of the gradient are repeated at multiples of the distance between the first and last color stop.

Further Reading: repeating-radial-gradient, repeating-linear-gradient

Last Word on Gradients

Browser support for gradients currently requires vendor-prefixing (which I'll cover later).

Advanced techniques such as radial gradients and repeating gradients do not enjoy the same level of support on all browsers.

Provide a sensible fall-back if gradients are not supported on the current platform.

Working with Text

Basic Introduction to Typography

Browsers define 5 basic generic font-families.

Serif Glyphs have finishing strokes, flared or tapering ends, or have actual serifed endings.
Sans-Serif Glyphs have stroke endings that are plain.
Cursive Glyphs in cursive fonts generally have either joining strokes or other cursive characteristics beyond those of italic typefaces. The glyphs are partially or completely connected, and the result looks more like handwritten pen or brush writing than printed letterwork.
Fantasy Fantasy fonts are primarily decorative fonts that contain playful representations of characters.
Monospace All glyphs have the same fixed width.

The actual default font used for each font-family is defined by the user agent.

One of these families should always be included in any font rule. 

a { font: 0.8em 'Segoe UI', Arial, sans-serif; }
Source: font-family

Specifying a Font

.example1 { font: 20px Verdana, sans-serif; }

Example 1: Verdana at 20px

Example 2: Fallback sans-serif at 20px 

.example3 { font: 0.8em/1.2em 'Segoe UI', Arial, sans-serif; }

Example 3: Segoe UI at 70% of the inherited size with 120% larger line-height 

.example4 { font: italic small-caps bold 0.8em/1.2em cursive; }

Example 4: Fallback cursive font in small-cap bold italic

Web Fonts

The days of sticking with "web safe" fonts are gone. Web fonts allow interesting typography without requiring images of text.

The "Fontspring @Font-Face Syntax" is currently the best way to include a web font to ensure it works on all platforms. 

@font-face {
   font-family: 'MyFontFamily';
   src: url('myfont-webfont.eot?#iefix') format('embedded-opentype'),
   url('myfont-webfont.woff') format('woff'),
   url('myfont-webfont.ttf')  format('truetype'),
   url('myfont-webfont.svg#svgFontName') format('svg');
}

Works in Safari 5, IE 6-9, FireFox 3.6+, Chrome 8+, iOS 3.2+, Android 2.2+, and Opera 11+

Source: The New Bulletproof @Font-Face Syntax

Vendor Prefixes

What's the Deal with Vendor Prefixes?

The CSS spec is a work-in-progress.

Experimental features are added using vendor prefixes.

This allows other browsers to safely ignore them.

Vendor prefixes are also used on CSS features that are not yet standardized.

Some prefixed features are safer to use than others.

FlexBox Example

The CSS working group decided to completely rewrite the Flexible Box Model spec.

This was unfortunately after it was already implemented in Chrome 4, Firefox 2, and IE 10...

Further Reading: Quick Hits with the Flexible Box Model, Learn You a Flexbox for Great Good!

Vendor Prefixing Best Practices

Include all code for all vendors and then future-proof with an un-prefixed version at the end. 

.myAwesomeGradient {
  background-image: -webkit-linear-gradient(top, greenyellow, green);
  background-image:    -moz-linear-gradient(top, greenyellow, green);
  background-image:     -ms-linear-gradient(top, greenyellow, green);
  background-image:      -o-linear-gradient(top, greenyellow, green);
  background-image:         linear-gradient(top, greenyellow, green);
}

The Vendor Prefixing Crisis... And It's All Your Fault

You should ALWAYS include all the vendor prefixes.

Site authors have a tendency to include only -webkit prefixes on many common CSS properties.

[The missing prefixed properties leads] to a reduced user experience on Opera and Firefox, which don't receive the same shiny effects such as transitions, gradients, and the like, even if the browser supported those effects.
Bruce Lawson, Opera

In light of this, Opera has decided to start implementing some -webkit prefixes.

The last time a single vendor emerged as the market leader and developers ignored other platforms was IE 6. Don't let IE 6 happen again.

Further Reading: W3C meeting minutes Feb. 7, 2012, The Impending CSS Vendor Prefix Catastrophe

SASS

What is SASS?

SASS is a CSS preprocessor that introduces variables, mixins, and selector nesting. 

/* SASS code */
$sassy: #BADA55;
a {
  color: $sassy;
  &:hover { color: lighten($sassy, 20%); }
  &:visited { color: darken($sassy, 20%); }
}
/* Generated CSS */
a { color: #bada55; }
a:hover { color: #dceca9; }
a:visited { color: #86a524; }
Further Reading: SASS

Variables

Variables in SASS can hold any value that's allowed by a CSS property. 

 /* SASS code */
$location: top;
$color: green;
$size: 45px;
$style: dashed;

/* Use #{} in selectors and property names */
.sassy-#{$location} {
  border-#{$location}: $size $style $color;
}
 /* Generated CSS */
.sassy-top {
  border-top: 45px dashed green;
}

Operations on Variables

Variables support standard math operations as well. 

/* SASS code */
$size: 20px;

div.sassy {
  height: $size * 4;
  width: $size / 2;
  padding: $size + 10px;
  margin: $size - 5px;
}
/* Generated CSS */
div.sassy {
  height: 80px;
  width: 10px;
  padding: 30px;
  margin: 15px;
}

Mixins

Mixins allow re-use of styles, properties, and selectors. 

/* SASS code */
@mixin blue-box {
  background-color: blue;
  border: 3px dashed invert(blue);
}

div.sassy {
  @include blue-box;
}
/* Generated CSS */
div.sassy {
  background-color: blue;
  border: 3px dashed yellow;
}

Making Generic Mixins

Arguments can be passed to mixins. 

/* SASS code */
@mixin styled-box($color, $size, $style: dashed) {
  background-color: $color;
  border: $size $style invert($color);
}

div.sassy {
  @include styled-box(blue, 3px);
}
/* Generated CSS */
div.sassy {
  background-color: blue;
  border: 3px dashed yellow;
}

Selector Nesting

Selector nesting allows you to write more specific selectors without all the copy and paste. 

/* SASS code */
.myClass {
  color: orange;
  div {
    color: green;
    #myBlueThing { color: blue; }
  }
}
/* Generated CSS */
.myClass { color: orange; }
.myClass div { color: green; }
.myClass div #myBlueThing { color: blue; }

Selector Nesting - The Parent Reference

Use & in a selector to reference the parent element. 

/* SASS code */
.myClass {
  color: blue;
  &:hover { color: green; }
  .ie8 & { color: red; }
  ul {
    .ie9 & li { color: orange; }
  }
}
 /* Generated CSS */
.myClass { color: blue; }
.myClass:hover { color: green; }
.ie8 .myClass { color: red; }
.ie9 .myClass ul li { color: orange; }

@extend

Use the @extend directive to build DRY styling rules 

/* SASS code */
.myClass {
  border: 1px dashed orange;
  background-color: yellow;
}

.otherClass {
  color: black;
  text-shadow: 1px 1px 3px gray;
}

.extendedClass {
  @extend .myClass, .otherClass;
  color: green;
  border-width: 3px;
}
/* Generated CSS */
.myClass, .extendedClass {
  border: 1px dashed orange;
  background-color: yellow;
}

.otherClass, .extendedClass {
  color: black;
  text-shadow: 1px 1px 3px gray;
}

.extendedClass {
  color: green;
  border-width: 3px;
}
Source: SASS Reference: extend

Reflows

Reflows are a triggered layout + repaint.

Video: Mozilla Home Page Reflow

What Triggers a Reflow?

Updating the position or size of any visible element will trigger an automatic layout and repaint.

  • Add, remove, update DOM nodes
  • Hiding a node with display: none
  • Animations
  • Adding a stylesheet
  • User resizing the viewport, zooming, scrolling

Reflow calculation is delayed as long as possible to batch changes together.

Source: How (not) to trigger a layout in WebKit for a comprehensive list of properties that trigger reflows in Webkit browsers.

How Can I Avoid Unnecessary Reflows?

Every page has at least one reflow (the initial layout).

The goal is to eliminate unnecessary reflows. 

/* Slow - 3 Reflows! */
var val = $('#item1').css('width'); // Read
$('#item1').css('width', val + 10); // Write
val = $('#item2').css('width');     // Read
$('#item2').css('width', val + 20); // Write
val = $('#item3').css('width');     // Read
$('#item3').css('width', val + 30); // Write
/* Fast - 1 Reflow */
var val1 = $('#item1').css('width'), // Read
    val2 = $('#item2').css('width'), // Read
    val3 = $('#item3').css('width'); // Read
$('#item1').css('width', val1 + 10); // Write
$('#item2').css('width', val2 + 20); // Write
$('#item3').css('width', val3 + 30); // Write
Further Reading: Rendering: repaint, reflow/relayout, restyle

Avoid Expensive CSS

Most CSS styles can be applied in a single pass.

Some require multiple passes or more expensive operations.

Expensive does not mean - do not use. It means use with restraint and common sense.

  • @font-face
  • box-shadow
  • opacity on ALL THE THINGS
  • gradients
  • text-align
Source: Avoiding Expensive CSS

Selector Performance

Selectors are evaluated right-to-left.

When trying to match an element against the thousands of potential matches in a stylesheet, the best approach is divide and conquer. Starting on the right allows elimination of the largest number of selectors.

Previously I mentioned all CSS rules are divided into 4 buckets.

  • Class - the right most portion of the rule is a class
  • ID - the right most portion of the rule is an ID
  • Tag - the right most portion of the rule is an element
  • Universal - Everything else

The fastest selectors search a single category.

The Universal category is the slowest of the four.

Source: Writing Efficient CSS

Descendant Selectors are Slow

After finding a candidate match with the right-most selector, you need to move up the DOM tree to test the other parts of the selector.

The performance is worse with very deep DOM trees.

The descendant selector is the most expensive selector in CSS. It is dreadfully expensive â€” especially if the selector is in the Tag or Universal Category.
Dave Hyatt, WebKit Architect 
/* Slooooowwww CSS */
div.myClass div div [type=text] {
/*...*/
}
<!-- Bad news HTML -->
<div class="myClass">
  <div class="not-myClass">
    <div class="not-myClass">
      <div class="not-myClass">
        <div class="not-myClass">
          <div class="not-myClass">
            <input type="text">

More on Descendant Selectors

Child selector always preferred over descendant selectors
.myClass > #item

When descendant selectors are required use the most specific category on the right (class or ID)
div .myClass

Remember all the SASS selector nesting niceties? Keep performance in mind when using them.
